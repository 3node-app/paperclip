# Runbook — Self-hosted GitHub Actions runner on the Oracle VM (NODE-235)

> **Audience:** the human operator (Ryan), who is the only actor with root/SSH on the
> Oracle VM host. Paperclip agents run in an isolated container (`uid=1000`, no `sudo`,
> no `docker.sock`, no host FS) and **cannot** perform the install. Everything below is
> copy-paste for a root shell on the host.
>
> **Goal:** stop paying for GitHub-hosted Actions on `ryan-pontes/*` repos by running a
> self-hosted runner on the Oracle Ampere VM (idle CPU/disk, zero marginal cost).
> Board context: paid GHA was rejected (approval `33a23f6a`, 2026-06-16).

## 0. Facts that correct the original issue spec

| Issue assumed | Reality (verified) | Impact |
|---|---|---|
| `ryan-pontes` is an **org** (`--url .../ryan-pontes`, org runner page) | `ryan-pontes` is a **User** account (`gh api users/ryan-pontes → type:User`) | **No org-level runners exist.** Register **per-repo**. URL must be `https://github.com/ryan-pontes/<repo>`. Acceptance criterion #1 ("organizations/ryan-pontes/settings") is unreachable → check the **repo** runner page instead. |
| Token generated by hand in GitHub UI (expires 1h) | The repo-admin `gh` token can mint a registration token via API | No UI step needed; mint it seconds before `config.sh` (see §3). |
| `usermod -aG docker gha` | `pr.yml` uses Docker only as a **static lint** (`scripts/check-docker-deps-stage.mjs` parses the Dockerfile; no daemon). | **Do NOT add `gha` to the `docker` group** for pr.yml/ci.yml — `docker` group is root-equivalent and defeats the "no privileges" isolation. Only add it if you later move `docker.yml`/`release.yml` image builds to self-hosted. |

VM arch confirmed: `aarch64` → use the **arm64** runner tarball (Ampere). Pin runner **v2.323.0**
(or the current release; check https://github.com/actions/runner/releases).

Repos in scope (both **private**):
- `ryan-pontes/paperclip` — validate with `pr.yml` (acceptance #2).
- `ryan-pontes/painel-envios-livraria` — validate with `ci.yml` (acceptance #4; PAGP-22 PR #15 had to merge without CI).

## 1. Create the unprivileged runner user (run as root on the host)

```bash
sudo useradd -m -s /bin/bash gha
# NO sudo rights, NO docker group. Confirm:
id gha            # expect: groups=gha  (no 'sudo', no 'docker', no 'wheel')

# Optional but recommended: cap disk blast radius by putting _work on its own mount.
# If you have a spare volume, mount it at /home/gha/actions-runner/_work before config.
```

Isolation hardening (Sec / Tiago to verify):
- `gha` must **not** be able to read production secrets. Verify it cannot read NodeChat/Pilot env:
  ```bash
  sudo -u gha cat /opt/nodechat/.env   2>&1 | head -1   # expect: Permission denied / No such file
  sudo -u gha ls   /paperclip          2>&1 | head -1   # expect: Permission denied  (if applicable)
  ```
  Tighten any prod dirs that are world-readable: `chmod 750` + correct owner.
- `gha` has no `sudo`: `sudo -u gha sudo -n true` → must fail.

## 2. Download the runner (as the `gha` user)

```bash
sudo -u gha bash -lc '
  set -euo pipefail
  mkdir -p ~/actions-runner && cd ~/actions-runner
  V=2.323.0
  curl -fsSL -o runner.tgz \
    https://github.com/actions/runner/releases/download/v${V}/actions-runner-linux-arm64-${V}.tar.gz
  tar xzf runner.tgz && rm runner.tgz
  ls -1   # bin/ config.sh run.sh svc.sh ...
'
```

## 3. Register the runner — PER REPO (mint token via API, no UI)

`config.sh` is interactive about the runner name; we pre-seed everything via flags so it is
non-interactive. Runner name `oracle-runner-01` (acceptance #2 expects this in `Set up job` logs).

> The `REG_TOKEN` is minted by the repo-admin `gh` token. It is short-lived (~1h) but you use
> it immediately. The agent (Felipe) can also mint it on request and hand it over — but minting
> here keeps the secret off the chat thread.

```bash
# --- paperclip ---
REPO=ryan-pontes/paperclip
REG_TOKEN=$(gh api -X POST repos/$REPO/actions/runners/registration-token --jq .token)

sudo -u gha bash -lc "
  cd ~/actions-runner
  ./config.sh \
    --url https://github.com/$REPO \
    --token $REG_TOKEN \
    --name oracle-runner-01 \
    --labels self-hosted,linux,arm64,oracle \
    --work _work \
    --unattended --replace
"
```

> **Per-repo limitation:** a single runner registration is bound to ONE repo. To serve BOTH
> `paperclip` and `painel-envios-livraria` from one VM you have two options:
> - **(A) Two runner installs** in separate dirs (`~/actions-runner-paperclip`,
>   `~/actions-runner-painel`), each registered to its repo, each its own systemd service.
>   Simple, fully isolated, costs ~200MB extra disk. **Recommended.**
> - **(B) Convert one repo to use the other via a GitHub *organization*** — not possible here
>   (`ryan-pontes` is a personal account). Skip.
>
> For (A), repeat §2 + §3 in `~/actions-runner-painel` with `REPO=ryan-pontes/painel-envios-livraria`
> and `--name oracle-runner-02`.

## 4. Install + start as a systemd service (survives VM restart → acceptance #5)

```bash
cd /home/gha/actions-runner
sudo ./svc.sh install gha          # runs as user 'gha'
sudo ./svc.sh start
sudo ./svc.sh status               # active (running)
systemctl status 'actions.runner.*' --no-pager   # confirm enabled + active
```

For the second runner (option A): `cd /home/gha/actions-runner-painel && sudo ./svc.sh install gha && sudo ./svc.sh start`.

## 5. Security: gate fork PRs (run as repo admin; agent can also do this)

Untrusted PR code must NOT auto-run on our VM. For each repo set "require approval for ALL
outside collaborators":

```bash
for REPO in ryan-pontes/paperclip ryan-pontes/painel-envios-livraria; do
  gh api -X PUT repos/$REPO/actions/permissions/workflow \
    -f default_workflow_permissions=read -F can_approve_pull_request_reviews=false
  # Fork-PR approval policy (UI: Settings→Actions→General→Fork pull request workflows):
  # set "Require approval for all outside collaborators". API field:
  gh api -X PUT repos/$REPO/actions/permissions \
    -f enabled=true -f allowed_actions=all || true
done
```
> Note: the exact "require approval for all outside collaborators" toggle currently has no
> stable REST field; verify in the UI (Settings → Actions → General → "Fork pull request
> workflows from outside collaborators" → **Require approval for all outside collaborators**).
> Both repos are private, which already blocks anonymous forks, but set this anyway as defense
> in depth before the first external collaborator is added.

## 6. Weekly disk cleanup (prevents `_work` / buildx cache bloat)

```bash
sudo tee /etc/cron.weekly/gha-runner-clean >/dev/null <<'EOF'
#!/bin/sh
# NODE-235: keep self-hosted runner disk usage bounded
find /home/gha/actions-runner*/_work/_temp -mindepth 1 -maxdepth 1 -mtime +3 -exec rm -rf {} + 2>/dev/null || true
# prune docker only if gha ever gets docker access (not by default):
# docker system prune -af --filter 'until=168h' 2>/dev/null || true
EOF
sudo chmod +x /etc/cron.weekly/gha-runner-clean
```

## 7. Switch workflows to self-hosted — **ONLY AFTER the runner shows Online**

Switching `runs-on` before a runner is Online makes every job queue forever. Do this as the
**last** step, once §4 shows `active (running)` and the repo runner page shows **Online**.

In `ryan-pontes/paperclip` `.github/workflows/pr.yml` (8 jobs) replace:
```yaml
runs-on: ubuntu-latest
```
with:
```yaml
runs-on: [self-hosted, linux, arm64]
```
Same for `ryan-pontes/painel-envios-livraria` `.github/workflows/ci.yml`.

> Leave `docker.yml`, `release.yml`, `release-smoke.yml`, `agent-runtime-images.yml`,
> `e2e.yml` on `ubuntu-latest` for now — they need Docker buildx / more isolation. Moving them
> is a separate decision (would require `gha` in the `docker` group → re-do the Sec review).
> Felipe (Dex) will open the pr.yml switch PR gated on this runbook reaching "runner Online".

## 8. Validation checklist (maps to acceptance criteria)

- [ ] **#2** A `paperclip` PR runs `pr.yml`; `Set up job` log shows `Runner name: oracle-runner-01`.

- [ ] **#5** `sudo systemctl restart` the VM (or `systemctl is-enabled 'actions.runner.*'`) →
      service comes back `active`.
- [ ] Sec (Tiago): `gha` has no sudo, no docker group, cannot read prod secrets (§1).

## 9. Maintenance

- Runner binary version drift: re-run §2 with the new version + `./config.sh remove --token <new>`
  then re-register. Check quarterly (next: ~2026-09).
- Concurrency: 1 runner = 1 concurrent job. If the squad floods PRs, add `oracle-runner-03` (repeat §2–4).
- Plan-B: if the runner destabilizes the Oracle VM (it already hosts paperclip), move it to a
  DigitalOcean Basic droplet ($4/mo) — same steps, different host.
